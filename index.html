🔴 交互与UI规范
1 立即配送时间显示变更
原：大約 17:01 › → 改：約 16:55 – 17:10 ›

展示格式：約 {HH:mm} – {HH:mm}，「約」字保留传达预估感
跨日处理：若时间横跨凌晨0点（即结束时间 < 开始时间），不再隐藏时间段，时间展示样式需与【预约单】现有的跨日时间段展示格式完全对齐（例如保持为预估 `HH:mm - HH:mm` 送达，或增加『次日』前缀等，与预约单跨日表现完全一致）。
降级退化（展示精确时间点）条件：
  1. 门店命中通用配置的 exclude_shop_ids 排除名单
  2. 当前时间（查询或下单请求时）不在通用配置的 active_time_periods 生效时段内
2 优享送时间配置逻辑（回归精确时间点）
业务决策：优享送在用户侧保持以【精确时间点】展示，不再改为时间段。但是其时间点的计算基准发生了变化。

计算公式：
计算基准时间 = 立配单区间终点（即 T + after_minutes）
优享送最终送达时间 = 计算基准时间 - 优享送提前分钟数

案例推演：
立配单预计 16:55–17:10 送达，优享送选择为提前 5 分钟。
则优享送预计送达时间点为：17:10 - 5 = 17:05。

💡 发单与展示时，系统根据此规则算出一个绝对的【单一精确时间点】传递给配送。所有用户侧、骑手侧以及短信通知的展示均完全按照此点位执行（如展示“预计17:05送达”）。
🔵 数据与算法逻辑
时间段计算公式
开始时间 = T_原始预计送达 - before_minutes
结束时间 = T_原始预计送达 + after_minutes

参数来源：通用配置接口返回字段 time_range_config
  before_minutes：默认 10（参考预约单前10后5，立配单可独立配置）
  after_minutes：默认 5

ℹ️ 原时间点 T 继续用于所有内部业务逻辑（商户出餐倒计时、骑手分配、小票打印、超时判定），仅用户端展示层改为时间段
功能开关（通用配置新增）
新增配置项 instant_order_time_range：
字段类型说明
enabledbool总开关
business_typesarray["restaurant","non_restaurant"]，按业务线控制
active_time_periodsarray生效时段，如[{"start":"11:00","end":"14:00"}]。为空数组则全时段生效
exclude_shop_idsarray排除门店ID列表（麦当劳等保持时间点）
before_minutesint前偏移分钟数，默认10
after_minutesint后偏移分钟数，默认5

✅ 优享送专项逻辑说明（回归全端精确点版）
优享送不再适用任何动态时间段展示方案。无论是用户界面的预估文案、订单详情文案，还是管理后台与短信分发，均纯粹地显示单独时间点。

📌 基准挂靠与系统下发规则（对齐立配单）：
1. 时间点偏移公式： 优享送送达时间点 = 立配时间段终点（即原始T + after_minutes） - 优享送配置的提前分钟数
2. 实例说明： 例如立配时间段预计为 16:55 – 17:10 生效期，优享送业务设为提前 5 分钟送达。
则下发给优享送订单各个模块的精准送达时间值就是 17:10 往前推 5 分钟 = 17:05。
3. 唯一基准： 订单中心由此公式算出一个确切的时间戳（即 17:05）后，将该时间单点统一下发。后续的超时判定、退款拦截以及所有的前端/骑手/后台展示都以此单一时间点为准。

* 备注：此前涉及的优享送时间段UI替换改造已舍弃。但因基准点顺延，仍存在多端联动的测试改动，请依下列细则验证。

🔴 配置规范
配置项与预约单的关系：
• 立配单和预约单使用各自独立的配置节点，互不影响
• 前后偏移量可各自独立配置，建议初始值：前10后5
• 排除门店名单也各自独立，立配单新增一条独立名单配置项

⚠️ 注意：参考预约单经验——配置中"前0后0"时，依然会显示时间段格式（段落样式），请运营在配置时确认前后偏移值不得同时为0
🔵 接口设计
通用配置接口 GET /api/common-config 返回新增节点：
instant_delivery_time_range: {
  enabled: true,
  business_types: ["restaurant","non_restaurant"],
  active_time_periods: [{ "start": "11:00", "end": "14:00" }], // 若为空数组或null表示全时段开启
  before_minutes: 10,
  after_minutes: 5,
  exclude_shop_ids: [1024, 1025, ...]
}

客户端每次进入下单流程时请求一次，结果可缓存至订单提交完成。
